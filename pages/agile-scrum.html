01 — THEORY

Understanding Agile & Scrum

Agile is a project management philosophy that values flexibility, collaboration, and delivering working software in short cycles. Scrum is the most popular Agile framework, organising work into time-boxed sprints (usually 1–4 weeks). Even if you're not a developer, understanding this vocabulary is essential for working in modern tech teams.

📋

Roles in Scrum

Scrum defines three key roles: the Product Owner (defines what to build and prioritises the backlog), the Scrum Master (facilitates the process and removes blockers), and the Development Team (builds the product).

"The Product Owner has prioritised the login feature for next sprint. The Scrum Master flagged a blocker with the API integration."
🔄

The Sprint Cycle

A sprint is a fixed time period (1–4 weeks) during which the team delivers a set of features. It starts with Sprint Planning, includes Daily Stand-ups, and ends with a Sprint Review and Retrospective.

"We committed to 8 story points this sprint. During the stand-up, João flagged that the payment integration is blocked by a third-party issue."
📊

Backlog & User Stories

The Product Backlog is a prioritised list of everything to be built. Items are written as User Stories: "As a [user], I want [feature] so that [benefit]". Each story has story points estimating effort.

"As a customer, I want to reset my password via email so that I can regain access to my account." — Estimated: 3 story points.
🚨

Common Agile Concepts

Key concepts include: MVP (Minimum Viable Product), Definition of Done (criteria for a task to be complete), Velocity (how much work a team completes per sprint), and Technical Debt (shortcuts that need fixing later).

"Our velocity has been averaging 25 points per sprint. We need to allocate time to address the technical debt in the authentication module."

Agile Communication Tips

In stand-ups, keep it to 3 points: what you did yesterday, what you'll do today, and any blockers.
Use "blocker" when something is preventing progress — it signals urgency to the team.
In retrospectives, focus on processes, not people: "What went well? What can we improve?"
Write user stories in the format: "As a [role], I want [feature] so that [benefit]."
02 — VOCABULARY

Useful Words & Phrases

Sprint
A time-boxed work cycle (1–4 weeks)
Backlog
Prioritised list of features and tasks
Stand-up
A brief daily meeting (15 min max)
User story
A feature described from the user's perspective
Story points
A unit estimating the effort of a task
Blocker
An obstacle preventing progress
MVP
Minimum Viable Product — simplest version
Velocity
Amount of work completed per sprint
Retrospective
Meeting to reflect on what to improve
Definition of Done
Criteria for a task to be considered complete
Technical debt
Shortcuts in code that need future fixing
Scrum Master
Person who facilitates the Scrum process
